# DungeonFighter v2 — Overview

DungeonFighter v2 is a turn-based RPG written in C# (.NET) with a data-driven combat/action system, character progression, and an Avalonia UI. Gameplay is built around **actions** (attacks, heals, effects) that are selected and executed in a structured combat flow, with systems for combo routing, thresholds, roll modifiers, and post-hit processing.

## Core gameplay pillars

- **Turn-based combat**: Entities take turns selecting and executing actions.
- **Actions as data**: Most combat behavior is configured through JSON/spreadsheet-driven action data that maps into runtime `Action` objects.
- **Combo system**: Actions can chain; combo routing can jump/skip/repeat via `ComboRouter`. **Combo damage AMP** uses each combo action’s **position in the hero’s strip** (first slot 1.00×, second slot the TECH baseline multiplier, then higher powers for later slots); opener/finisher tags do not reorder AMP relative to that visible sequence. Each **weapon type** (Sword, Dagger, Mace, Wand) has one **required basic** attack that must appear in the hero’s combo sequence while that weapon is equipped: it is defined in `Actions.json` with tag `**weapon_basic`** and a matching `**weaponTypes`** entry (`WeaponRequiredComboAction`, `ComboSequenceManager` ensure/inventory removal rules). Canonical basics: **STRIKE** (Sword), **STAB** (Dagger), **SLAM** (Mace), **CAST** (Wand); if no tagged row is found, those names are used as fallbacks. `**GearActionNames.Resolve`** always includes that required basic in the weapon’s resolved action-name list when absent (so starter/loot weapons that only list e.g. **JAB** still load **STRIKE** into the pool and the cannot-remove / re-inject rules apply). **Intelligence (effective INT) and the strip:** at `**GameConstants.ComboSequenceIntelligenceThreshold`** (10) or higher, which combo-strip action fires on a combo-path roll follows `**ComboStep`** in order, and `**ComboRouter`** rules apply after each successful combo hit. Below that threshold, combo-path strip indexing still follows `**ComboStep`** modulo strip length (same slot as the HUD highlight; heroes and enemies), while `**ComboStep**` advances **only to the next sequential slot**—authored `**ComboRouter`** branches (jump/skip/repeat/stop, etc.) are ignored (`Character.IncrementComboStep`). The Action Interaction Lab passes a deterministic salt into `ActionUtilities.ResolveComboStripIndex` so catalog picks align to a chosen lab d20. If the strip has **exactly one** combo action, `ResolveComboStripIndex` returns **0** without rolling (avoids an invalid `1d1` when strip size is 1). **Below combo threshold (named combo-strip swing vs synthetic normal):** the modified **d20** from the die (sheet roll modifications to the face only; not STR/INT/etc. roll bonuses) must meet the configured combo threshold (default 14; COMBO-type queued adjustments can still change that number), except on a natural 20. Queued **ACCURACY** shifts hit and post-roll **combo outcome** thresholds in combat, but does **not** lower the bar used to pick a named combo-strip action—so a low die cannot read as **roll: 9** yet execute a **14+** combo special. Otherwise `ActionSelector` uses an **unnamed synthetic normal** hit for both heroes and enemies—including enemies whose action pool is **only** combo-flagged specials, so they no longer fall through to the first special at low rolls. **Encounter boundaries:** `CombatStateManager.InitializeCombatEntities` calls `**Character.ResetCombo()`** on the hero and current foe so each new fight starts from the opener (step 0, combo mode off, baseline amp state); dungeon **room entry** (`RoomProcessor`) and **new dungeon** (`DungeonDisplayManager.StartDungeon`) also reset the hero’s combo so strip/UI never carries a prior room’s step into exploration.
- **Advanced mechanics**: Thresholds, roll modification, status effects, and multi-hit processing participate in the action execution pipeline. **Combat HUD thresholds** (left/right panels) preview queued FIFO/slot **ACCURACY** / **HIT** / **COMBO** shifts only; persistent stat/equipment roll bonus is **not** folded into those rows (it is added to attack total in `ActionExecutionFlow` / `HitCalculator`). Large deferred accuracy (e.g. wand **CAST** `heroAccuracy`) can still lower a *raw* preview below 1; the UI clamps the displayed ladder number to **1** so it stays readable, matching the minimum **1** floor applied when resolving effective threshold values for display.
- **Intelligence milestones (effective INT)**: INT no longer adds to the roll total. Instead, effective INT grants **milestone-based threshold bonuses** that shift **HIT / COMBO / CRIT** at specific breakpoints (per the designer table). These bonuses are applied after per-attack threshold reset so **action selection**, **combat resolution**, and the **HUD threshold preview** stay consistent.
- **Settings-driven tuning**: The settings UI persists changes to `GameSettings` and applies some updates immediately (e.g., action pool refresh) while other settings apply on next use.

## Key systems (high-level)

- **Google Sheets ↔ JSON (legacy/optional)**: The project has a full Sheets pull/push pipeline (weapons, mods, armor, suffixes, enemies, environments, dungeons) using `[GameData/SheetsConfig.json](GameData/SheetsConfig.json)` and `GameDataSheetsPullService` with `[JsonArraySheetConverter](Code/Data/JsonArraySheetConverter.cs)`. **OAuth push** (`GameDataSheetsPushService`, `SheetsPushConfig.json`) can **skip individual spreadsheet tabs**: per-tab flags (`pushActionsTab`, `pushWeaponsTab`, …) default to true for older configs; Settings → Balance Tuning exposes checkboxes so e.g. **ENEMIES** can be left unchecked to avoid overwriting that tab while pushing other data. **If you are no longer using Sheets**, the game still runs fully from the JSON files under `GameData/`, and the Sheets-related helper scripts can be treated as legacy tooling.
- **Loot — three-category item prefixes**: Loot rolls at most one **Adjective**, one **Material**, and one **Quality** prefix per item (`Modification.prefixCategory`; omitted = Adjective for legacy rows in `Modifications.json`). **When `TuningConfig.json` does not override a rarity**, **Common** fills one random slot among the three; **Uncommon** fills two random distinct slots; **Rare+** fills all three. When **`itemAffixByRarity`** supplies a row for that rarity, **prefix slot count** (0–3) is rolled between the configured **min** and **max** using **extra chance** (`ItemAffixByRaritySettings`, `LootBonusApplier`), then categories are chosen as today for that count. Display order is **Quality → Adjective → Material**, then the base name, then modification names starting with `of `, then stat-bonus suffixes (`ItemPrefixHelper`, `ItemGenerator.GenerateItemNameWithBonuses`, `ItemNameParser`). **Affix-line rarity (prefix + stat suffix):** for each prefix slot and each rolled stat-suffix line, the game rolls a tier (**Common** … **Mythic**) using **`RarityTable.json`** row **weights** (`RarityData`), then picks **uniformly** among catalog rows in that tier: prefixes match **`Modification.ItemRank`** (blank `ItemRank` is treated as **Common** only); stat suffixes match **`StatBonus.Rarity`** (blank / missing → **Common** after load). If the weighted tier has no rows in that pool, the roller retries up to a small cap over tiers that do have rows. **Quality** tiers use `Effect` `gearPrimaryStatMultiplier` (rolled multiplier applied once per piece to weapon `BaseDamage+BonusDamage` in `WeaponItem.GetTotalDamage` and to armor totals in `HeadItem`/`ChestItem`/`FeetItem.GetTotalArmor`). **Material** rows use `equipmentStr` / `equipmentAgi` / `equipmentTec` / `equipmentInt` (DEX-style materials map to **TEC**) aggregated in `EquipmentBonusCalculator` with suffix stat bonuses. Extra pools live in `GameData/PrefixMaterialQuality.json`, merged at load with `Modifications.json` in `LootDataCache`. `RarityTable.json` **`Modifications`** counts no longer drive prefix count (prefixes follow rarity name as above); stat/action bonus columns still seed defaults when tuning omits a rarity. Stat suffix and action bonus **counts** also roll min→max when tuning defines extra chance. **Divine** (`Effect` `reroll`) is stripped after roll and replaced by a higher-tier adjective re-roll (`LootBonusApplier.ResolveRerollAdjectiveIfPresent`). Tests: `ItemPrefixHelperTests`, `LootBonusApplierTests`; contextual loot counts **adjective** mods only for theme bias.
- **Data tags (actions, weapons, armor, enemies)**: **ACTIONS** — row-2 header `**TAGS`** holds an optional comma/semicolon list of extra tokens (ingested by `[SpreadsheetActionDataCsvParser](Code/Data/SpreadsheetActionDataCsvParser.cs)`); **category** and **rarity** are still merged into runtime `ActionData.Tags` separately. Place the column wherever row-1/row-2 labels resolve; **columns E–F** on the live Actions tab remain **push-skipped** for on-sheet formulas (`SheetsPushUtilities`), so do not put author-owned TAGS data in E–F unless you migrate that contract. **Weapons / armor** — optional JSON property `**tags`** (array of strings) in `Weapons.json` / `Armor.json` and matching sheet columns; `[ItemGenerator](Code/Data/ItemGenerator.cs)` copies normalized tags onto generated `[Item](Code/Items/Item.cs)` instances. **Enemies** — optional `**tags`** on each row in `Enemies.json` / ENEMIES sheet. Shared normalization: `[GameDataTagHelper](Code/Data/GameDataTagHelper.cs)`.
- **Enemy attributes from `Enemies.json`**: `baseAttributes` and `growthPerLevel` use nullable numeric fields per stat. Shipped enemy rows list **all four** base combat stats (STR/AGI/TEC/INT) so the ENEMIES sheet and HUD stay in sync with designer intent. If a row still omits keys, omitted stats are **not** treated as zero; `EnemyStatCalculator` fills them from tuning **baseline × archetype**. Per-level attribute gains come only from `**growthPerLevel`**, normalized so STR+AGI+TEC+INT growth sums to **6** each level. Explicit `0` in JSON still means zero for that cell when present.
- **ENEMIES sheet column layout (canonical, 17 columns A–Q)**: **A** `name`, **B** `archetype`, **C–F** base attributes (`strength`, `agility`, `technique`, `intelligence`), **G–J** growth per level (same four), **K** `baseHealth`, **L** `healthGrowthPerLevel`, **M** `actions`, **N** `isLiving`, **O** `description`, **P** `colorOverride`, **Q** `tags` (optional JSON array in the cell, e.g. `["undead","boss"]`). Row 1 is the **category band** (`base attributes`, `growth`, …); row 2 is the short field names (see `JsonArraySheetConverter`). The game expects enemies in the canonical nested JSON shape (`baseAttributes` + `growthPerLevel`) produced by the spreadsheet pull/push pipeline (no legacy root stat keys or legacy overrides).
- **Actions spreadsheet (`GameData/Actions.json`)**: Rows can be tagged for **non-hero** pools. Use **category** `ENEMY` with **tags** including `enemy` for enemy-only attacks (so loot/weapon resolution does not assign them to the hero). Use **category** `ENVIRONMENT` with **tags** including `environment` plus a **lowercase dungeon theme** (e.g. `forest`, `crypt`); optional `room-{type}` in tags scopes data. Environment hazards use spreadsheet **target** `AOE`. Runtime room loading filters environment actions by theme (or the `generic` tag). Environmental rows use the legacy `EnvironmentalActions.json` **id** as the spreadsheet **action** key (**uppercase**, words separated by spaces; room-scoped legacy ids omit the old `room_` prefix) so every row stays unique.
- **New game starter weapons**: When any `Weapons.json` rows include the **starter** tag, the pick-one-weapon step lists those rows in file order (`StarterCatalogItems.ResolveStarterWeaponMenuCatalogRows`, `GameInitializer.BuildStarterWeaponsForMenu` / `CreateStarterWeaponForMenuIndex`). Otherwise it falls back to **one tier-1** row per class path in `ClassPresentationConfig.ClassWeaponOrder` (Mace, Sword, Dagger, Wand): the **first tier-1** row of that `Type` in list order (`WeaponTypeFromCatalog.TryGetFirstTierOneCatalogRow`). The shipped catalog defines **four** starter-tagged choices (one per weapon type). `GameData/StartingGear.json` carries **starting armor only**.
- **Settings system**: Avalonia settings panels save through an orchestrator and apply via a single apply service. See `Documentation/05-Systems/SETTINGS_SYSTEM_ARCHITECTURE.md`. **Game Settings — tags:** the **Actions** tab shows a read-only **runtime tags** line for the selected action (same merged tag list as the form, including category/rarity). **Items** (weapons and armor) shows **Tags:** on each row, comma-separated **Tags** in **Edit Item**, and persists `tags` in `Weapons.json` / `Armor.json` on **Save**. **Enemies** is its own category: catalog from `Enemies.json` with editable comma-separated tags; **Save** writes the file and calls `EnemyLoader.LoadEnemies()` so combat uses the new tags.
- **Settings tool — Item Generation Lab**: New settings tab **Item Generation** can force-generate a batch (default 100) of weapons/armor from the current data + loot rules while forcing **rarity** (or **Any**), **tier**, **weapon type**, and/or **armor slot**, then sorts best→worst using **rarity → tier → primary stats**. When **Rarity** is **Any**, the panel exposes **fixed rarity chances** (Common→Mythic) that the lab uses for its base rarity roll **regardless of hero level** (then normal “rarity required by bonuses” rules can still step an item up). The panel also exposes a **Base ratio** input: when set, fixed chances are derived as a geometric series where each rarity is **X× rarer** than the previous (Common X× Uncommon, Uncommon X× Rare, …), then normalized to 100%. **Hero level** still matters for the **live loot** rarity preview line and for tier rolls. **Dungeon Δ** is editable too (relative to hero): effective dungeon level = `hero + Δ` (clamped 1–99). When **Tier** is **Any**, generation rolls tier using hero-vs-dungeon **loot level** (`LootTierCalculator`, same as `LootGenerator`) and the UI shows the matching `TierDistribution.json` percentages for that loot level. The same tab edits **per-rarity affix bounds** for real loot and the lab: **minimum** prefix slots / stat suffixes / action bonuses, optional **extra %** (per-step chance toward a **max**), stored in `TuningConfig.json` under `itemAffixByRarity` (`ItemAffixPerRarityEntry`, `LootBonusApplier` rolls counts on each drop). The UI presents these as **Prefix / Suffix / Actions** (player-facing labels): **Prefix** = Quality/Adjective/Material slots (0–3), **Suffix** = extra **StatBonuses.json** suffix lines (often “of …”), **Actions** = extra granted-action lines — distinct from prefix-slot modifications that can also change stats. Implementation: `ItemGenerationSettingsPanel` + `ItemGenerationPanelHandler` + `ItemGenerationLabService` (`GenerateBatch`, `SortBestToWorst`). **Layout:** root **Grid** uses **Auto** for the control stack and **`*`** for **Generated Items** | **Selected Item** so the batch list owns vertical space. The top area stacks **Generation Parameters** (`2×4` grid + **live loot rarity** preview), the **lab rarity** + **live loot tier** strip, **Clear** + **Generate**, and **Affix bounds** in a collapsed-by-default **Expander** (affix grid + notes in an inner **ScrollViewer** with **MaxHeight** so tuning does not consume the whole window). The list uses an internal **ListBox** scroll. Item display names use idempotent assembly (`ItemGenerator.GetBaseItemName` strips already-applied prefix/suffix tokens, including **repeated passes** for stat suffixes and `of …` modification tails so equal-length pairs like **of the Titan** + **of Alacrity** do not leave the inner suffix on the base name) so repeated bonus application does not stack the same prefix or suffix twice.
- **Inventory (Avalonia)**: On the **inventory** screen, the top **action strip** mirrors the hero’s combo sequence (same order as the right-hand SEQUENCE panel). **Right-click** a filled strip card to remove that slot—same rules as removing from the sequence list (`InventoryComboManager` / `cpi:rm:N`, including **weapon-required** basic protection). Drag reorder on the strip remains available when combo reorder is allowed (not during an active dungeon run). Equip / compare / trade-up prompts ignore strip removes until you finish or cancel those flows. The right-hand **POOL** panel lists gear/class actions under **(from gear)** and bag item actions under **INVENTORY (from bag)**; clicking an inventory row **equips** that item (swapping the correct slot, previous piece returns to the bag) and **adds** that action to the sequence when it appears as a combo action in the pool (`cpi:invpool:N`, `InventoryActionPoolEntries`, `InventoryComboManager`). **Pointer hover** on a bag row (center list) or on **left-panel** gear/stats while inventory is open shows the same detail tooltip band as elsewhere, including a resolved **Actions:** name list for that item (`LeftPanelTooltipBuilder`, `MouseInteractionHandler`, `InventoryScreenRenderer` clickables). **`GearActionNames.Resolve`** is the single source for those names. Armor with an explicit **`GearAction`** plus affixes no longer had that name appended twice (legacy **`GetArmorExtraActionNames`** re-added the same **`GearAction`** after **`Resolve`** had already included it), which caused misleading **Actions:** lines like **Tight Combo, Tight Combo**. Weapons can still list the same name twice when **`GearAction`** and **`ActionBonuses`** intentionally grant duplicate pool entries.
- **Dungeon selection (Avalonia)**: Choosing **Custom difficulty** shows the level you are typing **on the same row** as that menu entry (`Level:` + digits + `_` cursor), aligned with the other dungeon lines. Validation and short hints use the bottom **non-blocking** message strip (`ShowInvalidKeyMessage`) instead of `ShowMessage`, which cleared the canvas and drew centered text over the OPTIONS block.
- **Action execution**: Action execution is orchestrated and delegated to specialized executors and processors (attack/heal, multi-hit, etc.). See `Documentation/01-Core/ARCHITECTURE.md` for file entry points.
- **Character effects**: Character/actor state includes transient “next action” modifiers (speed/damage/amp/multihit) that should affect the following action execution.

### Poison, burn, and bleed (DoT)

- **Poison**: Cumulative **percent of max HP** (`PoisonPercentOfMaxHealth`). Successful applications **add** to that %; the global DoT tick deals `floor(maxHP × % / 100)` and **does not** reduce stored %. **Cleanse** can shave stored % by a small fixed step. Tuning uses a shared **5s game-time** tick interval (`poison.tickInterval` in `TuningConfig.json`) with burn. **Weapon `weaponPoison` mods** roll a per-crit % from each mod’s `MinValue`/`MaxValue` in `Modifications.json` (e.g. Venomous **1–2%**, Venomous Blade **2–3%** on a **critical hit** only); those rolls sum for the delta applied on that crit. Poison, burn, and bleed from **weapon stats** or weapon `**StatusEffects`** apply only when `CombatEvent.IsCritical` is true. **Actions** with built-in `CausesPoison` / `CausesBurn` / `CausesBleed` and **no** spreadsheet **trigger conditions** follow the same crit-only rule; if the action defines **trigger conditions** (`ONHIT`, `ONCRITICAL`, etc.), those conditions gate the whole status bundle as before. Application runs once inside `ActionExecutionFlow`; the colored UI path must only **display** `StatusEffectMessages` from that run (re-applying would double the debuff).
- **Burn**: **Intensity** plus **pending from hits**; each global tick deals damage equal to current intensity, then decays intensity by 1 and merges pending (same pattern for processing order in code).
- **Bleed**: Same **intensity + pending** model as burn, but **one pulse when the afflicted actor finishes their combat turn** (not on the global poison/burn tick). Weapon mods use deterministic `**weaponPoison` / `weaponBurn` / `weaponBleed`** values on **critical hits** only (no random proc rolls for those).
- **Combat log spacing**: Global poison/burn tick output is recorded as a `PoisonDamage` UI block with the **afflicted** actor’s name. `TextSpacingSystem` uses that so a tick on the hero does not insert an extra blank line before the hero’s following combat action (the prior line’s “last attacker” would otherwise be the enemy).
- **Combat log phrasing**: The **hits** keyword on damage lines (and the **hits** suffix in multi-hit counts like `(3 hits)`) is drawn in **white**; critical/combo/damage emphasis stays on action names and damage numbers (`CombatColorStrategy.GetHitsColor`, `DamageFormatter.FormatDamageDisplayColored`).

## Current focus area (for this change)

### Data — tags on actions, items, and enemies

- **Actions:** CSV/sheet import reads `**TAGS`** into spreadsheet rows; loot/action code already filters pools using runtime tags (e.g. excluding `**environment`** / `**enemy`** from hero gear random actions). **Settings UI** surfaces merged runtime tags for the selected action (see Settings system bullet above).
- **Items:** Catalog `tags` on weapon/armor JSON rows propagate to loot-generated `Item.Tags` for future criteria (filters, bonuses). **Settings UI** lists and edits those tags on the Items tab and in the item edit dialog.
- **Enemies:** `EnemyData.Tags` is first-class JSON for sheet pull/push and validators; combat features can read tags when needed. **Settings UI** lists enemies and supports tag edits with JSON save + loader refresh.

### Saves — death tombstones (not loadable)

When the hero dies, the game persists a **tombstone** JSON (`isDead: true`) under `GameData/character_{id}_dead.json` (or `character_save_dead.json` for legacy single-slot saves without an ID) and **removes** the live `character_*_save.json` / `character_save.json` file. `ListAllSavedCharacters` and `LoadCharacterAsync` ignore tombstones so fallen adventurers do not appear in the load list. **Clear all saves** in settings also deletes `character_*_dead.json` and the legacy dead file. Death persist runs from `GameScreenCoordinator.ShowDeathScreen` (and the console `DeathScreenHandler` fallback), wrapped in try/catch so a disk error cannot block the death UI.

### Progression — XP to next level and mid-run level-ups

- **XP bar** (`CharacterProgression.GetXpRequiredToAdvanceFromLevel`): one **tier-1 dungeon completion** (same 10× formula as `XPRewardSystem.AwardDungeonCompletionXP` at dungeon level **1**) is the unit. The bar is **that many completions**: **1** for L1→2 (so the first dungeon run finishes a level if completion XP is the main grant), then **1.5**, **2**, **3**, **4**, **5**, … for higher levels so it takes more runs to level over time. `Progression.BaseXPToLevel2` > **0** scales the whole curve so the L1→2 bar equals that value; `Progression.XPScalingFactor` (when > **0**) multiplies every bar as a global pace knob (≤ **0** ⇒ **1.0**). Helpers: `GetBaseXpForContentLevel`, `GetStandardDungeonCompletionXpForLevel`, `GetExpectedDungeonsToLevelFromLevel`.
- **Immediate XP** (`CharacterFacade.AddXP`): progression already increments `Level` per threshold; reward application uses `LevelUpWithInfo(levelAlreadyIncremented: true)` and passes the **per-step** level (`oldLevel + step + 1`) for messages, dungeon completion entries, and class points so multi-threshold grants stay consistent.

### Combat — combo reset at encounter / room / dungeon boundaries

- **Behavior:** Hero (and current enemy) combo routing state resets whenever a combat encounter is initialized, when the player enters a new dungeon room, and when a new dungeon run starts—so the first meaningful combo-path swing of a new fight is never still keyed off the previous encounter’s strip position or combo-mode carryover.
- **Code:** `CombatStateManager.InitializeCombatEntities`, `CombatManager.RunCombat` (start/end uses `ResetCombo` for the player), `RoomProcessor.ProcessRoom`, `DungeonDisplayManager.StartDungeon`; test `CombatStateManagerTests.TestInitializeCombatEntitiesResetsPlayerAndEnemyCombo`.

### Combat — hero debuffs and DoT between encounters

- **Behavior:** After each combat (`CombatManager.RunCombat`), after each survived room (`RoomProcessor`), and when a dungeon finishes or the player exits early (`DungeonOrchestrator`), the hero’s `ClearAllTempEffects()` runs so poison/burn/bleed, stun/weaken/slow/shields, crit-miss penalty, advanced stack effects (vulnerability, fortify, silence, temp HP, etc.), and character-side combo/roll/next-attack buffers do not carry into the next fight or the overworld.
- **Implementation:** `Actor.ClearAllTempEffects` clears every combat status field on `Actor`; `Character.ClearAllTempEffects` still clears `CharacterEffects` and temp stat bonuses. `CharacterFacade.ClearAllTempEffects` delegates to `Character.ClearAllTempEffects` (single path). Tests: `ActorClearTempEffectsTests` in the entity test runner.

### Data / Google Sheets — item suffixes (`StatBonuses.json`)

- **Pull**: `statBonusesSheetUrl` in `SheetsConfig.json` (published CSV or same-doc `edit?gid=…` link as other tabs).
- **Push**: `statBonusesSheetTabName` in `SheetsPushConfig.json` (default `**SUFFIXES`** when the field was missing/blank). Balance Tuning can store the tab **gid** in the new **SUFFIXES → StatBonuses.json** field; push still requires a tab whose **title** matches `statBonusesSheetTabName`.
- **Columns (canonical / JSON):** `Name`, `Description`, `Value`, **`Rarity`** (affix pool tier for loot rolls), `StatType`, optional `ItemRank` (Action Lab catalog filter: blank = visible under every rarity filter). Sheets import still accepts a legacy **`Weight`** column; it is removed and **`Rarity`** defaults to **Common** when no `Rarity` cell was present (`JsonArraySheetConverter.NormalizeStatBonusesJsonArrayRow`).

### Action Interaction Lab — controls pop-out layout

The **Action Lab** auxiliary window lists the current **foe**, **HP**, a scrollable **enemy type** picker (▲ types / ▼ types), then turn info (**Next** actor, **Actions taken** counter, strip step), d20 grid, and the scrollable **action catalog**. **Actions taken** increments for each interactive **[ Step ]** that records a lab turn (same as undo history), adds the sum of resolved combat turns across encounters after a **[ Sim N ]** batch finishes, and resets when the lab clears fight history (gear apply/clear, enemy type swap, enemy level rebuild, or the tools-panel **[ Reset ]** control). **[ Reset ]** also clears the center combat log, refills both fighters to full HP, clears status and temporary combat effects, and returns both combo strip pointers to the first slot while **keeping** equipped items, the combo strip contents, and the current enemy encounter. **Back** (undo) removes the last interactive step from that count. The enemy-type list shows **five** visible rows (was two) so more foes are visible at once; the catalog uses the remaining vertical space above the fixed footer.

When the lab starts, window **placement** (sizes unchanged) uses the monitor that contains the main game window: the main window is **centered** in that display’s **working area**, and the lab tools window is **right-aligned** inside the same working area with a small margin so it stays on-screen on smaller displays. If a pop-out **Game Settings** window is already open on that same monitor, it is **nudged to the left** inside the working area so the three-window layout matches a typical wide-desktop arrangement (`ActionLabWindowPlacement`, `ActionLabControlsWindow`).

### Combat right panel — location names

In the **LOCATION** block, **Dungeon** and **Room** values (and the enemy name line when combat is active) ellipsize only when wider than the panel’s inner monospace width (`LayoutConstants.RIGHT_PANEL_WIDTH` minus content inset), not a fixed 20-character cap—so names like **Tomb of the forgotten** render in full.

### Combat right panel — status effects (enemy)

The **right panel** status-effects block labels **Enemy:** active DoT/debuff lines from `StatusEffectDisplayLines`. When an enemy template is **non-living** (`isLiving` false in `Enemies.json` → `Enemy.IsLiving`), the game does not apply **bleed** or **poison % DoT** ticks to that foe; the panel shows a cyan line **Immune: Bleed, Poison** under **Enemy:** so it matches combat rules (distinct from **IsAlive**, which is HP > 0).

### Action Interaction Lab — gear editor (right-click equipment)

Right-clicking a gear row in lab combat opens the **Action Lab — weapon / armor** dialog. You pick a base row from data, then **multiple** optional **prefix** rows (modifications) and **multiple** optional **suffix** rows (stat bonuses); each list is multi-select (toggle rows on or off). Apply builds the item with every selected prefix and suffix (rolled mod values still use each template’s min/max range). The dialog opens at about **1000×560** client size so lists are not cramped, is **resizable**, and the three scrollable lists **split extra height equally** so you see more rows without only tiny fixed caps. The **weapon** dialog adds **Weapon type**, **Tier**, and **Rarity** dropdowns: type narrows the weapon list (Mace, Sword, Dagger, Wand); **tier** matches the weapon row’s numeric tier (choices come from `Weapons.json`); rarity narrows **prefixes** by each row’s `ItemRank`, **suffixes** by optional `ItemRank` on `StatBonuses.json` entries (blank `ItemRank` keeps a suffix visible for every rarity), and **weapons** by a tier→rarity band (T1 Common … T5 Legendary, with **Mythic** / **Transcendent** still showing tier-5 bases). **Head, body, and feet** use the same rarity behavior on base armor, prefixes, and suffixes, plus **Armor class** and **Tier** (tier choices from `Armor.json` for that slot): classes are the armor display name with the last word removed (e.g. `Cloth Cap` → `Cloth`, `Studded Leather Boots` → `Studded Leather`). The SUFFIXES sheet / JSON canonical columns include optional `**ItemRank`** alongside `Name`, `Description`, `Value`, `**Rarity**`, `StatType`.

### Action Interaction Lab — enemy level (right panel)

The **Action Interaction Lab** is a stepped combat sandbox (cloned hero, lab enemy, forced catalog action, fixed or random d20). The combat **right panel** shows the current foe (HP, combat stats, thresholds). This change adds:

- **Enemy level** on the right panel in lab combat (`Lvl N` plus **(Δ)** vs the lab hero: `Lvl 9 (-2)` when the hero is 11; Δ = enemy level − hero level), styled like the hero’s level on the left.
- **Hero level** (left panel `Lvl`): left-click / right-click adjusts the hero by one step as before, and applies the **same** level delta to the lab enemy when the hero’s level actually changes (clamped 1–99), so both stay aligned from the hero row. **Enemy level** on the right panel still changes **only** the enemy (independent of the hero).
- **Left / right click** on the right-panel **enemy** level row to raise or lower enemy level (1–99) alone. The lab **rebuilds** the enemy at the new level (loader enemies via `EnemyLoader.CreateEnemy`; default dummy via `TestCharacterFactory` with per-level HP/damage/armor/speed scaling), then **clears step history** and **re-bootstraps** combat so encounter state stays consistent.
- **Undo / replay** and **batch simulation** snapshots carry the effective enemy level (`LabCombatSnapshot.EnemyLevel`, `CaptureSimulationSnapshot`).
- **Direct-stat dummies**: `Enemy.GetEffectiveStrength()` returns base **Damage** (so damage math and the DMG line match); the panel also shows **attack speed**.

### HUD — left panel hero name (class identity)

When the composed HUD class title is still the configured **default no-points** label (typically **Fighter**—no weapon-path class points yet, or the highest path is still below the first tier gate), the hero **name** stays **plain white**. After that title advances to a real solo, duo, or multi-path class string, the name is drawn as **per-letter color cycles** using a palette keyed off the dominant **weapon path** (solo) or the **top two paths** by class points (hybrids), aligned with multiclass title composition (`HeroNamePanelColoredText`, `CharacterPanelRenderer`).

### MultiHit Mod (Hero base stats → next action) — prior milestone

The settings UI exposes **Hero base stats** including action speed %, action damage %, **MultiHit mod**, and amp mod %. Those modifiers apply to the hero’s **next** executed action; prior work ensured MultiHit mod flows through data, execution, and tests.
