# DungeonFighter v2 — Overview

DungeonFighter v2 is a turn-based RPG written in C# (.NET) with a data-driven combat/action system, character progression, and an Avalonia UI. Gameplay is built around **actions** (attacks, heals, effects) that are selected and executed in a structured combat flow, with systems for combo routing, thresholds, roll modifiers, and post-hit processing.

- **Room search loot (post-combat)**: Searching a cleared room no longer drops random **equipment**. Successful search rolls pick one row from **`Consumables.json`** (editable in Google Sheets via the **CONSUMABLES** tab; **PULL** via `SheetsConfig.json` → `consumablesSheetUrl`; **PUSH** via `SheetsPushConfig.json` → `consumablesSheetTabName` / `pushConsumablesTab`). Items are **eaten or drunk immediately** (no bag entry): food restores a flat or **% of max HP** heal; potions grant STR, AGI, TEC, INT (**% of base stats** from the sheet) or flat threshold shifts for HIT, COMBO, CRIT, and CRIT MISS. Potion effects stack in `Character.DungeonSearchBuffs` until the dungeon run ends (start of run, completion, early exit, or clone-after-death clears them); threshold shifts re-apply after each combat threshold reset alongside gear.

## Core gameplay pillars

- **Turn-based combat**: Entities take turns selecting and executing actions.
- **Actions as data**: Most combat behavior is configured through JSON/spreadsheet-driven action data that maps into runtime `Action` objects.
- **Combo system**: Actions can chain; combo routing can jump/skip/repeat via `ComboRouter`. **Combo damage AMP** uses each combo action’s **position in the hero’s strip** (first slot 1.00×, second slot the INT baseline multiplier, then higher powers for later slots); opener/finisher tags do not reorder AMP relative to that visible sequence. Each **weapon type** (Sword, Dagger, Mace, Wand) has one **required basic** attack that must appear in the hero’s combo sequence while that weapon is equipped and the matching weapon-path class is still below its first named tier: it is defined in `Actions.json` with tag `**weapon_basic`** and a matching `**weaponTypes`** entry (`WeaponRequiredComboAction`, `ComboSequenceManager` ensure/inventory removal rules). Canonical basics: **STRIKE** (Sword), **STAB** (Dagger), **SLAM** (Mace), **CAST** (Wand); if no tagged row is found, those names are used as fallbacks. `**GearActionNames.Resolve`** always includes that required basic in the weapon’s resolved action-name list when absent (so starter/loot weapons that only list e.g. **JAB** still load **STRIKE** into the pool and the pre-class cannot-remove / re-inject rules apply). **Intelligence (effective INT) and the strip:** at `**GameConstants.ComboSequenceIntelligenceThreshold`** (10) or higher, which combo-strip action fires on a combo-path roll follows `**ComboStep`** in order, and `**ComboRouter`** rules apply after each successful combo hit. Below that threshold, combo-path strip indexing still follows `**ComboStep`** modulo strip length (same slot as the HUD highlight; heroes and enemies), while `**ComboStep**` advances **only to the next sequential slot**—authored `**ComboRouter`** branches (jump/skip/repeat/stop, etc.) are ignored (`Character.IncrementComboStep`). The Action Interaction Lab passes a deterministic salt into `ActionUtilities.ResolveComboStripIndex` so catalog picks align to a chosen lab d20. If the strip has **exactly one** combo action, `ResolveComboStripIndex` returns **0** without rolling (avoids an invalid `1d1` when strip size is 1). **Below combo threshold (named combo-strip swing vs synthetic normal):** the modified **d20** from the die (sheet roll modifications to the face only; not STR/INT/etc. roll bonuses) must meet the configured combo threshold (default 14; COMBO-type queued adjustments can still change that number), except on a natural 20. Queued **ACCURACY** shifts hit and post-roll **combo outcome** thresholds in combat, but does **not** lower the bar used to pick a named combo-strip action—so a low die cannot read as **roll: 9** yet execute a **14+** combo special. Otherwise `ActionSelector` uses an **unnamed synthetic normal** hit for both heroes and enemies—including enemies whose action pool is **only** combo-flagged specials, so they no longer fall through to the first special at low rolls. **Encounter boundaries:** `CombatStateManager.InitializeCombatEntities` calls `**Character.ResetCombo()`** on the hero and current foe so each new fight starts from the opener (step 0, combo mode off, baseline amp state); dungeon **room entry** (`RoomProcessor`) and **new dungeon** (`DungeonDisplayManager.StartDungeon`) also reset the hero’s combo so strip/UI never carries a prior room’s step into exploration.
- **Advanced mechanics**: Thresholds, roll modification, status effects, and multi-hit processing participate in the action execution pipeline. **Combat HUD thresholds** (left/right panels) preview queued FIFO/slot **ACCURACY** / **HIT** / **COMBO** shifts only; persistent stat/equipment roll bonus is **not** folded into those rows (it is added to attack total in `ActionExecutionFlow` / `HitCalculator`). Large deferred accuracy (e.g. wand **CAST** `heroAccuracy`) can still lower a *raw* preview below 1; the UI clamps the displayed ladder number to **1** so it stays readable, matching the minimum **1** floor applied when resolving effective threshold values for display. Clicking the visible left-panel **THRESHOLDS** rows toggles between those d20 ladder numbers and exclusive d20 outcome chances for **Crit**, **Combo**, normal **Hit**, **Miss**, and **Crit Miss**.
- **Intelligence milestones (effective INT)**: INT no longer adds to the roll total. Instead, effective INT grants **milestone-based threshold bonuses** that shift **HIT / COMBO / CRIT** at specific breakpoints (per the designer table). These bonuses are applied after per-attack threshold reset so **action selection**, **combat resolution**, and the **HUD threshold preview** stay consistent.
- **Settings-driven tuning**: The settings UI persists changes to `GameSettings` and applies some updates immediately (e.g., action pool refresh) while other settings apply on next use.
- **Balance tuning automation**: Profile-driven 3-step CLI — `TUNESIM` / `TUNEANALYZE` / `TUNEAPPLY`. Profiles in `GameData/TuningProfiles/` control simulation scope and analysis validators/suggesters. `TUNEPROFILES` lists options. Legacy: `LEVELSIM` etc. Skill: `.cursor/skills/level-balance-tuning/SKILL.md`.

## Key systems (high-level)

- **Google Sheets ↔ JSON (legacy/optional)**: The project has a full Sheets pull/push pipeline (weapons, mods, armor, suffixes, enemies, environments, dungeons) using `[GameData/SheetsConfig.json](GameData/SheetsConfig.json)` and `GameDataSheetsPullService` with `[JsonArraySheetConverter](Code/Data/JsonArraySheetConverter.cs)`. **OAuth push** (`GameDataSheetsPushService`, `SheetsPushConfig.json`) can **skip individual spreadsheet tabs**: per-tab flags (`pushActionsTab`, `pushWeaponsTab`, …) default to true for older configs; Settings → Balance Tuning exposes checkboxes so e.g. **ENEMIES** can be left unchecked to avoid overwriting that tab while pushing other data. **If you are no longer using Sheets**, the game still runs fully from the JSON files under `GameData/`, and the Sheets-related helper scripts can be treated as legacy tooling.
- **Loot — affix counts by item slot + combo strip cap:** `TuningConfig.json` → `itemAffixByRarity` supports **`perItemType`** (`Head`, `Chest`, `Legs`, `Feet`, `Weapon`) overrides in addition to flat **`perRarity`**; `LootBonusApplier` resolves counts using the concrete `Item.Type`. Each rarity row can tune **extra combo strip slots** from the affix pass (`extraComboSlots`, `extraComboSlotsMax`, `extraComboSlotsExtraChance` on `ItemAffixPerRarityEntry`) the same way as prefix/suffix/action lines: min → optional stepped rolls to max, with magic find scaling on those extra chances; the rolled amount is **added** to `Item.ExtraActionSlots` after the catalog `extraActionSlots` / min–max roll. Settings → Item Generation affix grid includes an **Extra slots** column group (+%, Min, Max). Hero combo length is capped by **`LootSystemConfig.ComboSequenceBaseMax`** / **`ComboSequenceAbsoluteMax`** plus extra slots from **all equipped gear** (catalog **`Item.ExtraActionSlots`**, loot affix pass above, suffix mechanics typed **`ExtraActionSlots`**, rolled once from **`extraActionSlots`** or inclusive **`extraActionSlotsMin`** / **`extraActionSlotsMax`** on **Armor.json** and **Weapons.json** rows via `ItemGenerator.RollCatalogExtraActionSlots`); `ComboSequenceMaxHelper` / `ComboSequenceManager` / `CharacterFacade` enforce the cap, inventory UI surfaces “sequence full,” and gear changes trim removable strip actions without dropping the required weapon basic. **Armor.json** rows can carry optional base combat fields (STR/AGI/TEC/INT, HIT/COMBO/CRIT, optional catalog **`attackSpeed`** → **`Item.CatalogAttackSpeed`** summed into equipment **`AttackSpeed`** for swing-time reduction, `extraActionSlots` / min/max pair, `minActionBonuses`) that copy onto generated armor and feed `EquipmentBonusCalculator` / generation floors for action-bonus lines on head items.
- **Loot — weapon vs armor slot mix:** `LootItemSelector.RollLootCategory` rolls **50% weapon** vs **50% armor**; armor then picks **head**, **chest**, **legs**, or **feet** at **12.5%** each before choosing a catalog row for the resolved tier (`LootItemSelector`, `LootGenerator`, with tier/slot fallbacks when a pool is empty).
- **Loot — three-category item prefixes**: Loot rolls at most one **Adjective**, one **Material**, and one **Quality** prefix per item (`Modification.prefixCategory`; omitted = Adjective for legacy rows in `Modifications.json`). **When `TuningConfig.json` does not override a rarity**, **Common** fills one random slot among the three; **Uncommon** fills two random distinct slots; **Rare+** fills all three. When **`itemAffixByRarity`** supplies a row for that rarity, **prefix slot count** (0–3) is rolled between the configured **min** and **max** using **extra chance** (`ItemAffixByRaritySettings`, `LootBonusApplier`), then categories are chosen as today for that count. Display order is **Quality → Adjective → Material**, then the base name, then modification names starting with `of `, then stat-bonus suffixes (`ItemPrefixHelper`, `ItemGenerator.GenerateItemNameWithBonuses`, `ItemNameParser`). **Affix-line rarity (prefix + stat suffix):** for each prefix slot and each rolled stat-suffix line, the game rolls a tier (**Common** … **Mythic**) using **`RarityTable.json`** row **weights** (`RarityData`), then picks **uniformly** among catalog rows in that tier: prefixes match **`Modification.ItemRank`** (blank `ItemRank` is treated as **Common** only); stat suffixes match **`StatBonus.Rarity`** (blank / missing → **Common** after load). If the weighted tier has no rows in that pool, the roller retries up to a small cap over tiers that do have rows. **Quality** tiers use `Effect` `gearPrimaryStatMultiplier` (rolled multiplier applied once per piece to weapon `BaseDamage+BonusDamage` in `WeaponItem.GetTotalDamage` and to armor totals in `HeadItem`/`ChestItem`/`LegsItem`/`FeetItem.GetTotalArmor`). **Material** rows use designer-facing stat names (`STRENGTH` / `AGILITY` / `TECHNIQUE` / `INTELLIGENCE`) or legacy internal keys (`equipmentStr` / `equipmentAgi` / `equipmentTec` / `equipmentInt`) aggregated in `EquipmentBonusCalculator` with suffix stat bonuses. Extra pools live in `GameData/PrefixMaterialQuality.json`, merged at load with `Modifications.json` in `LootDataCache`. `RarityTable.json` **`Modifications`** counts no longer drive prefix count (prefixes follow rarity name as above); stat/action bonus columns still seed defaults when tuning omits a rarity. Stat suffix and action bonus **counts** also roll min→max when tuning defines extra chance. **Divine** (`Effect` `reroll`) is stripped after roll and replaced by a higher-tier adjective re-roll (`LootBonusApplier.ResolveRerollAdjectiveIfPresent`). Tests: `ItemPrefixHelperTests`, `LootBonusApplierTests`, `EquipmentSystemTests`; contextual loot counts **adjective** mods only for theme bias.
- **Data — PREFIX / modifications sheet (columns A–I):** Canonical tab layout for prefix rows (see `JsonArraySheetConverter.ModificationsCanonicalHeaders`): **A** `DiceResult`, **B** `ItemRank`, **C** `Name`, **D** `Description`, **E** `Effect`, **F** `value` (import maps to both `MinValue` and `MaxValue`; legacy data and ranged rolls can keep `MinValue`/`MaxValue` in JSON—push may emit them as extra columns after **I** when they differ), **G** `prefixCategory`, **H** `ATTRIBUTE REQUIREMENT`, **I** `REQUIREMENT VALUE` (same stat abbrev / canonicalization as weapons and armor). JSON may also use an `attributeRequirements` object on a `Modification`. On generated gear, prefix gates merge with the base item’s catalog requirements (`Item.CatalogAttributeRequirements`, `Item.RecomputeAttributeRequirementsIncludingModifications`) so equip checks and tooltips stay correct after rerolls. CSV import runs `GameDataJsonNormalizer.NormalizeModificationsImportRow`; tests: `JsonArraySheetConverterTests.ModificationsCsvImportValueAndAttributeRequirementColumns`, `ItemTests` (recompute).
- **Data tags (actions, weapons, armor, enemies)**: Canonical vocabulary (77 registry tags) lives in `TagDefinitions.cs` — pool gates (`environment`, `enemy`, `weapon`, `class`, `item`, `action`, `unique`, `starter`), elements, environment structure/activity (`elegant`, `dilapidated`, `dormant`, `cycling`, `active`), mechanic tags (`swift`, `bludgeon`, `focus`, `insight`), roll tags (`confidence`, `footwork`, `target`, `aim`), and action routing (`required`, `opener`, `finisher`). Combat applies mechanic tags as next-action bonuses and roll tags as threshold shifts; rooms can set `unstableThresholdMod` and structure tags adjust d20 rolls by hero level. See `Documentation/TAG_REGISTRY.md`. **ACTIONS** — row-2 header `**TAGS`** holds an optional comma/semicolon list of extra tokens (ingested by `[SpreadsheetActionDataCsvParser](Code/Data/SpreadsheetActionDataCsvParser.cs)`); **category** and **rarity** are still merged into runtime `ActionData.Tags` separately.
- **Enemy attributes from `Enemies.json`**: `baseAttributes` and `growthPerLevel` use nullable numeric fields per stat. Shipped enemy rows list **all four** base combat stats (STR/AGI/TEC/INT) so the ENEMIES sheet and HUD stay in sync with designer intent. If a row still omits keys, omitted stats are **not** treated as zero; `EnemyStatCalculator` fills them from tuning **baseline × archetype**. Per-level attribute gains come only from `**growthPerLevel`**, normalized so STR+AGI+TEC+INT growth sums to **6** each level. Explicit `0` in JSON still means zero for that cell when present.
- **ENEMIES sheet column layout (canonical, 21 columns A–U)**: **A** `region`, **B** `biome`, **C** `location`, **D** `rarity`, **E** `name`, **F** `tags`, **G** `archetype`, **H–K** base attributes (`strength`, `agility`, `technique`, `intelligence`), **L–O** growth per level (same four), **P–Q** `HEALTH` band (`baseHealth`, `healthGrowthPerLevel`), **R** `actions`, **S** `isLiving`, **T** `description`, **U** `colorOverride`. Row 1 is the **category band** (`base attributes`, `growth`, `HEALTH`); row 2 is the short field names (see `JsonArraySheetConverter`). **Spawn tier roll** (after dungeon `possibleEnemies` filter): default 50% Common, 10% Uncommon+biome, 15% Uncommon+region, 15% Uncommon+location, 5% Rare+location, 5% anywhere — per settlement type (Rural / Town / City) via **Settings → Enemy tuning** and `EnemySystemConfig.SpawnTierWeightsBySettlement`. Settlement type comes from room location name (`city` / `town` / `village`), optional `settlementType` on travel regions in `Regions.json`, or tuning overrides. See `EnemySpawnFilter` / `SettlementTypeResolver`. Rarity also scales gold/XP via `EnemyRarityRewardHelper`.
- **Actions spreadsheet (`GameData/Actions.json`)**: Rows can be tagged for **non-hero** pools. Use **category** `ENEMY` with **tags** including `enemy` for enemy-only attacks (so loot/weapon resolution does not assign them to the hero). Use **category** `ENVIRONMENT` with **tags** including `environment` plus a **lowercase dungeon theme** (e.g. `forest`, `crypt`); optional `room-{type}` in tags scopes data. Environment hazards use spreadsheet **target** `AOE`. Runtime room loading filters environment actions by theme (or the `generic` tag). Environmental rows use the legacy `EnvironmentalActions.json` **id** as the spreadsheet **action** key (**uppercase**, words separated by spaces; room-scoped legacy ids omit the old `room_` prefix) so every row stays unique.
- **New game — Training Ground (pre-weapon)**: After creating a new hero (`MainMenuHandler` / `CharacterManagementHandler`), the game offers **Training Ground** before weapon selection: **1** enters a code-only dungeon (`GameConstants.TrainingGroundDungeonName`, `PreWeaponTrainingFlow`, `Dungeon.Generate`) with **one** hostile room and a **Training Dummy** (13 HP, 1 damage, ~8s base attack cadence, direct-stat `Enemy`); **2** skips with no XP/loot. The hero has **no equipped weapon** for that run; `ActionLoader.ReloadActions` + `CharacterSerializer.RebuildCharacterActions` supply default/class/fallback actions so combat works. **Unarmed tutorial baseline:** the default combo action is **PUNCH HARD** (**100%** damage, **100%** speed). Training Dummy combat scripts the hero's tutorial d20 sequence as **10 hit → 4 miss → 14 combo → 1 critical miss → 20 crit combo**; the 13 HP dummy survives the hit/combo lesson and dies on the final crit combo. Completing the run skips the normal dungeon completion screen, shows the short path intro line **"The quest to find yourself begins when you choose a path..."** shimmering between warm white and cool white (`PreWeaponPathIntroRenderer`, `PreWeaponPathIntroHandler`), then any key opens weapon selection (`DungeonRewardManager` / `DungeonOrchestrator`); choosing **Skip tutorial** also shows that path intro before weapon selection. Death in the tutorial revives and returns to the offer (`GameScreenCoordinator.ShowDeathScreen`). `Character.PendingPreWeaponTrainingGround` + save field `pendingPreWeaponTrainingGround` persist mid-flow. Automated AI skips the offer and continues through the path intro when reached (`ClaudeAIGamePlayer`). **Avalonia:** encounter combat runs on a threadpool task; center-panel `DisplayTiming` must invoke scheduled renders only **after** releasing its lock (nested `ScheduleRender` / `ForceRender` from a paint callback otherwise deadlocks the UI), and `HeroActionStripFeedback` must marshal timer + canvas invalidate to the UI thread so the first swings in Training Ground do not freeze the client.
- **New game starter weapons**: When any `Weapons.json` rows include the **starter** tag, the pick-one-weapon step lists those rows sorted by class weapon path (Barbarian→Warrior→Rogue→Wizard), then order within the starter-tagged subset; **at most one menu row per weapon type**—the first starter-tagged row for each `Type` wins if duplicates exist (`StarterCatalogItems.ResolveStarterWeaponMenuCatalogRows`, `GameInitializer.BuildStarterWeaponsForMenu` / `CreateStarterWeaponForMenuIndex`). Otherwise it falls back to **one tier-1** row per class path in `ClassPresentationConfig.ClassWeaponOrder` (Mace, Sword, Dagger, Wand): the **first tier-1** row of that `Type` in list order (`WeaponTypeFromCatalog.TryGetFirstTierOneCatalogRow`). The shipped catalog defines **four** starter choices (one per weapon type). `GameData/StartingGear.json` may list **optional starting armor** (default **empty**); armor can also be added per slot via **starter**-tagged rows in `Armor.json`. New heroes equip only the chosen **weapon**; **bag inventory** starts empty.
- **Settings system**: Avalonia settings panels save through an orchestrator and apply via a single apply service. See `Documentation/SETTINGS_SYSTEM_ARCHITECTURE.md`. **Game Settings — tags and starter/action visibility:** the **Actions** tab shows a read-only **runtime tags** line for the selected action (same merged tag list as the form, including category/rarity), editable **Assign to Weapon Types** checkboxes, and a **Required Weapon Basic** checkbox that persists the `weapon_basic` tag used by `WeaponRequiredComboAction` (e.g. **STRIKE** + Sword, **STAB** + Dagger, **SLAM** + Mace, **CAST** + Wand). **Items** (weapons and armor) shows **Tags:** and **Starter:** on each row; weapon rows also show resolved **Actions:** using `GearActionNames.ResolveWeaponType`, so starter weapon choices reveal the action rules they will grant. **Edit Item** exposes comma-separated **Tags** plus a first-class **Starter** checkbox and persists `tags` in `Weapons.json` / `Armor.json` on **Save**. **Enemies** is its own category: catalog from `Enemies.json` with editable comma-separated tags; **Save** writes the file and calls `EnemyLoader.LoadEnemies()` so combat uses the new tags.
- **Settings tool — Item Generation Lab**: New settings tab **Item Generation** can force-generate a batch (default 100) of weapons/armor from the current data + loot rules while forcing **rarity** (or **Any**), **tier**, **weapon type**, and/or **armor slot**, then sorts best→worst using **rarity → tier → primary stats**. When **Rarity** is **Any**, the panel exposes **fixed rarity chances** (Common→Mythic) that the lab uses for its base rarity roll **regardless of hero level** (then normal “rarity required by bonuses” rules can still step an item up). The panel also exposes a **Base ratio** input: when set, fixed chances are derived as a geometric series where each rarity is **X× rarer** than the previous (Common X× Uncommon, Uncommon X× Rare, …), then normalized to 100%. **Hero level** still matters for the **live loot** rarity preview line and for tier rolls. **Dungeon Δ** is editable too (relative to hero): effective dungeon level = `hero + Δ` (clamped 1–99). When **Tier** is **Any**, generation rolls tier using hero-vs-dungeon **loot level** (`LootTierCalculator`, same as `LootGenerator`) and the UI shows the matching `TierDistribution.json` percentages for that loot level. The same tab edits **per-rarity affix bounds** for real loot and the lab: **minimum** prefix slots / stat suffixes / action bonuses / **extra strip slots**, optional **extra %** (per-step chance toward a **max**), stored in `TuningConfig.json` under `itemAffixByRarity` (`ItemAffixPerRarityEntry`, `LootBonusApplier` rolls counts on each drop). The UI presents these as **Prefix / Suffix / Actions / Extra slots** (player-facing labels): **Prefix** = Quality/Adjective/Material slots (0–3), **Suffix** = extra **StatBonuses.json** suffix lines (often “of …”), **Actions** = extra granted-action lines, **Extra slots** = additional `Item.ExtraActionSlots` from the affix pass (additive to catalog) — distinct from prefix-slot modifications that can also change stats. Implementation: `ItemGenerationSettingsPanel` + `ItemGenerationPanelHandler` + `ItemGenerationLabService` (`GenerateBatch`, `SortBestToWorst`). **Layout:** root **Grid** uses **Auto** for the control stack and **`*`** for **Generated Items** | **Selected Item** so the batch list owns vertical space. The top area stacks **Generation Parameters** (`2×4` grid + **live loot rarity** preview), the **lab rarity** + **live loot tier** strip, **Clear** + **Generate**, and **Affix bounds** in a collapsed-by-default **Expander** (affix grid + notes in an inner **ScrollViewer** with **MaxHeight** so tuning does not consume the whole window). The **affix tuning grid** uses proportional column widths so +%/Min/Max fields grow with the panel; **lab fixed rarity** and **live tier** previews sit in a fixed-width (**340px**) strip beside that matrix so wide windows do not leave a dead zone to the right and those controls stay on-screen. The list uses an internal **ListBox** scroll. Item display names use idempotent assembly (`ItemGenerator.GetBaseItemName` strips already-applied prefix/suffix tokens, including **repeated passes** for stat suffixes and `of …` modification tails so equal-length pairs like **of the Titan** + **of Alacrity** do not leave the inner suffix on the base name) so repeated bonus application does not stack the same prefix or suffix twice.
- **Inventory (Avalonia)**: On the **inventory** screen, the top **action strip** mirrors the hero’s combo sequence (same order as the right-hand SEQUENCE panel). The **next** swing slot (`ComboStep`) uses a **white** border; each resolved hero action pulses that panel **red** on miss, **green** on a normal hit, and **gold** only when a **multi-step** combo completes (last strip slot or a tagged **finisher**), using the same on/off half-blink timing (miss uses miss duration; hits use success duration; defaults **1200 ms** / **3000 ms**, half-blink **150 ms**; tune under **Settings → Gameplay**). `HeroActionStripFeedback` drives combat + inventory rendering. Pressing numpad **`*`** on the main inventory screen auto-equips currently empty gear slots from bag items in inventory order, skipping occupied slots and any item blocked by the normal attribute requirement gate. **Action hover tooltips** (combat strip, inventory strip, and pool rows via `CombatActionStripBuilder`) call out **Opener** / **Finisher** when the action is tagged for first/last combo slots. Strip cards use a **red** action name for the data-driven required basic (`WeaponRequiredComboAction`) instead of repeating **Weapon basic** prose; tooltips and summaries also omit the **Type \| Target \| combo** line and **Ability**-cadence attack bonus lines so slots stay readable. Strip panels beyond the character’s **effective max combo length** (`ComboSequenceMaxHelper.GetEffectiveMax`: loot base/absolute cap plus equipped **ExtraActionSlots**) draw a **black** border so unused placeholders visually disappear against the background while the fixed minimum panel count keeps layout stable. **Right-click** a filled strip card to remove that slot—same rules as removing from the sequence list (`InventoryComboManager` / `cpi:rm:N`, including pre-class **weapon-required** basic protection). Drag reorder on the strip remains available when combo reorder is allowed (not during an active dungeon run). Equip / compare / trade-up prompts ignore strip removes until you finish or cancel those flows. The right-hand **POOL** panel lists gear/class actions under **(from gear)** and bag item actions under **INVENTORY (from bag)**; clicking an inventory row **equips** that item (swapping the correct slot, previous piece returns to the bag) and **adds** that action to the sequence when it appears as a combo action in the pool (`cpi:invpool:N`, `InventoryActionPoolEntries`, `InventoryComboManager`) **only if** the hero’s **effective** STR/AGI/TEC/INT meet the item’s catalog **`attributeRequirements`** (merged from `Weapons.json` / `Armor.json` via `GameDataJsonNormalizer` + `ItemGenerator` onto `Item.AttributeRequirements`). Otherwise equip is blocked with a clear message (`Character.TryEquipItem`, `InventoryItemActionHandler`, comparison pre-check, Action Lab `TryApplyLabGear` + `ShowInvalidKeyMessage`). Tooltips and item-stat panels show a **Requires:** summary line when requirements exist (`Item.GetAttributeRequirementsSummaryLine`, `LeftPanelTooltipBuilder`, `ItemStatsFormatter`). **Pointer hover** on a bag row (center list) or on **left-panel** gear/stats while inventory is open shows the same detail tooltip band as elsewhere, including a resolved **Actions:** name list for that item (`LeftPanelTooltipBuilder`, `MouseInteractionHandler`, `InventoryScreenRenderer` clickables). **`GearActionNames.Resolve`** is the single source for those names. Armor no longer synthesizes a random action merely because it has prefix or stat-suffix affixes; random granted-action lines now come from explicit **`GearAction`** or rolled **`ActionBonuses`** governed by the Item Generation **Actions** +%/Min/Max table. Armor with an explicit **`GearAction`** plus affixes no longer had that name appended twice (legacy **`GetArmorExtraActionNames`** re-added the same **`GearAction`** after **`Resolve`** had already included it), which caused misleading **Actions:** lines like **Tight Combo, Tight Combo**. Weapons can still list the same name twice when **`GearAction`** and **`ActionBonuses`** intentionally grant duplicate pool entries.
- **Inventory bag action uniqueness**: The **INVENTORY (from bag)** action list collapses duplicate action names across compatible bag items, keeping the first matching bag item as the click/equip target. The equipped **POOL (from gear)** list is intentionally unchanged, so duplicates still appear when multiple equipped items grant the same action.
- **Dungeon selection (Avalonia)**: Choosing **Custom difficulty** shows the level you are typing **on the same row** as that menu entry (`Level:` + digits + `_` cursor), aligned with the other dungeon lines. Validation and short hints use the bottom **non-blocking** message strip (`ShowInvalidKeyMessage`) instead of `ShowMessage`, which cleared the canvas and drew centered text over the OPTIONS block.
- **Dungeon exit prompt (Avalonia)**: The between-room exit choice menu is framed by the standard cyan divider both **above** `1 - Stay and continue through the dungeon` and below the two choices, so the prompt is visually bounded before the next **ENTERING ROOM** header renders. The room-cleared message leaves spacing only and does not add its own plain divider, avoiding a duplicate separator directly above the prompt.
- **Combat log (Avalonia)**: With the pointer over the framed **center column below the action strip**, the **mouse wheel** scrolls the same display buffer as arrow keys / page keys (`MainWindow` → `MouseInteractionHandler.HandlePointerWheelChanged` → `CanvasUICoordinator.ScrollUp` / `ScrollDown`). ### Poison, burn, bleed, and acid (DoT)

- **Poison**: Cumulative **percent of max HP** (`PoisonPercentOfMaxHealth`). Successful applications **add** to that %; the global DoT tick deals `floor(maxHP × % / 100)` and **does not** reduce stored %. **Cleanse** can shave stored % by a small fixed step. Tuning uses a shared **5s game-time** tick interval (`poison.tickInterval` in `TuningConfig.json`) with burn and acid. **Weapon `weaponPoison` mods** roll a per-crit % from each mod’s `MinValue`/`MaxValue` in `Modifications.json` (e.g. Venomous **1–2%**, Venomous Blade **2–3%** on a **critical hit** only); those rolls sum for the delta applied on that crit. Poison, burn, bleed, and acid from **weapon stats** or weapon `**StatusEffects`** apply only when `CombatEvent.IsCritical` is true. **Actions** with built-in `CausesPoison` / `CausesBurn` / `CausesBleed` / `CausesAcid` and **no** spreadsheet **trigger conditions** follow the same crit-only rule; if the action defines **trigger conditions** (`ONHIT`, `ONCRITICAL`, etc.), those conditions gate the whole status bundle as before. Application runs once inside `ActionExecutionFlow`; the colored UI path must only **display** `StatusEffectMessages` from that run (re-applying would double the debuff).
- **Burn**: **Intensity** plus **pending from hits**; each global tick deals damage equal to current intensity, then decays intensity by 1 and merges pending (same pattern for processing order in code).
- **Bleed**: Same **intensity + pending** model as burn, but **one pulse when the afflicted actor finishes their combat turn** (not on the global poison/burn/acid tick). Weapon mods use deterministic `**weaponPoison` / `weaponBurn` / `weaponBleed` / `weaponAcid`** values on **critical hits** only (no random proc rolls for those).
- **Acid**: Same **intensity + pending** model and **global tick** as burn. Each tick deals intensity HP (armor-bypass like other DoTs) and adds that amount to cumulative `AcidArmorReduction`, which reduces effective flat armor for subsequent attacks (heroes via `GetEffectiveArmor`, enemies via `DamageCalculator.ResolveTargetArmor`). Works on **non-living** foes (unlike bleed/poison). Sample loot prefix **Caustic** (`weaponAcid` 2–3) in `Modifications.json`. Existing **Expose** stack/turn effect is unchanged.
- **Combat log spacing**: Global poison/burn/acid tick output is recorded as a `PoisonDamage` UI block with the **afflicted** actor’s name. `TextSpacingSystem` uses that so a tick on the hero does not insert an extra blank line before the hero’s following combat action (the prior line’s “last attacker” would otherwise be the enemy).
- **Combat log phrasing**: The **hits** keyword on damage lines (and the **hits** suffix in multi-hit counts like `(3 hits)`) is drawn in **white**; critical/combo/damage emphasis stays on action names and damage numbers (`CombatColorStrategy.GetHitsColor`, `DamageFormatter.FormatDamageDisplayColored`).

### Audio — cues, music, and the Audio settings tab

Cross-platform audio is implemented in `Code/Audio/` on top of the **SoundFlow** NuGet package (MiniAudio backend, MIT, runs on Windows / macOS / Linux without extra installs). The system separates **music** (one streaming track at a time, crossfaded on state changes) from **SFX** (short, cached, fire-and-forget). Out-of-the-box decoders cover **WAV / MP3 / FLAC**; **OGG** requires `SoundFlow.Extensions.FFmpeg` (not bundled).

- **`AudioCue` enum** lists every named trigger (e.g. `Music_MainMenu`, `Music_Combat`, `Menu_Select`, `Menu_Back`, `Menu_Invalid`, `Combat_CriticalMiss`, `Combat_Miss`, `Combat_Hit`, `Combat_ComboComplete`, `Combat_CriticalHit`, `Combat_HeroHurt`, `Combat_EnemyDied`, `Combat_HeroLowHealth`, `Combat_EnemyLowHealth`, `Progression_LevelUp`, `Loot_ItemPickup`, `Loot_Equip`, `Loot_Save`, `Dungeon_Enter`, `Dungeon_RoomClear`). Each cue is mapped at runtime to an audio file by **`GameData/Audio/AudioConfig.json`** (master / music / sfx volume, master / music / sfx enabled, `cueMap` of cue → `{ file, volume, rateLimitMs? }`, `stateMusicMap` of `GameState` → music cue). Master, music, and SFX mute state now all live in `AudioConfig`; muting music stops only music while SFX remain governed by the SFX mute.
- **`ActionEventPublisher` + `AudioCueDispatcher`** route combat audio. Direct action outcomes fire immediately from `ActionEventPublisher` through `AudioCues.Trigger(...)`: critical miss (`Combat_CriticalMiss`), miss (`Combat_Miss`), hero-offense hit (`Combat_Hit`), hero-offense combo (`Combat_ComboComplete`), hero-offense critical hit (`Combat_CriticalHit`), and enemy successful hit against the hero (`Combat_HeroHurt`). `AudioCueDispatcher` still subscribes to non-outcome `CombatEventBus` events (`EnemyDied`, `HeroLowHealth`, `EnemyLowHealth`, `StatusEffectApplied`) and rate-limits chatty cues (`Combat_Hit`, `Combat_HeroHurt`, `Progression_XpGain` default ~80 ms). Low-health events fire when the hero or current enemy crosses from above 20% HP to at-or-below 20% HP and remains alive, including action damage and poison/burn/bleed ticks. **`MusicController`** subscribes to `GameStateManager.StateChanged` and crossfades music when the active `GameState` changes (e.g. `MainMenu` → `Dungeon` → `Combat` → `DungeonCompletion`).
- **Music crossfades are gapless:** incoming and outgoing music players are restarted by the game audio engine after `PlaybackEnded` instead of relying on SoundFlow's recursive `IsLooping` end-of-stream path. If there is no outgoing track, the incoming cue still fades in over the configured crossfade duration instead of starting at full volume. If the outgoing track reaches its natural end while it is still fading out, the backend restarts it from the beginning and keeps applying the fade curve until the transition is finished, preventing silence during long crossfades or short music stingers without re-entering SoundFlow EOF handling.
- **Non-DI call sites** fire cues through the static **`AudioCues.Trigger(cue)`** facade — menus (`MenuCommand`, `MainMenuHandler` invalid-input branch, `MainWindowInputHandler` Esc/Back), progression (`LevelUpManager`), inventory (`InventoryManager.AddItem`, `EquipmentManager.EquipItem` / `UnequipItem`), dungeon flow (`DungeonDisplayManager.StartDungeon`, room clear path in `DungeonRunnerManager`), and save (`CharacterSaveManager`).
- **Settings → Audio** (new tab in `SettingsPanel.axaml`) edits `AudioConfig.json` with live-applied master/music/SFX volume + mute, state→music dropdowns, and a scrollable cue row grid: each row has **Browse…** (Avalonia `OpenFileDialog` rooted at `GameData/Audio/`), volume slider, **Test** (plays the cue right now through a dedicated preview path that bypasses mute/zero-volume state), and **Clear**. Normal gameplay audio respects `AudioConfig.masterEnabled`, `musicEnabled`, and `sfxEnabled` independently, so the old gameplay `EnableSoundEffects` setting no longer suppresses SFX behind the Audio tab's music mute.
- **Engine abstraction**: `IAudioEngine` with `Play`, `PlayMusic`, `StopMusic`, `SetBusVolume`, `Shutdown`. `SoundFlowAudioEngine` is the production backend; `NullAudioEngine` is used by tests and any headless run (CI, automated tuning, MCP). Missing files never crash — the dispatcher logs once and skips playback.

### Progression — XP to next level and mid-run level-ups

- **XP bar** (`CharacterProgression.GetXpRequiredToAdvanceFromLevel`): one **tier-1 dungeon completion** (same 10× formula as `XPRewardSystem.AwardDungeonCompletionXP` at dungeon level **1**) is the unit. Helpers: `GetBaseXpForContentLevel`, `GetStandardDungeonCompletionXpForLevel`, `GetExpectedDungeonsToLevelFromLevel`.
- **Immediate XP** (`CharacterFacade.AddXP`): progression already increments `Level` per threshold; reward application uses `LevelUpWithInfo(levelAlreadyIncremented: true)` and passes the **per-step** level (`oldLevel + step + 1`) for messages, dungeon completion entries, and class points so multi-threshold grants stay consistent.
- **Class upgrade action slots:** Each weapon-path class tier reached via `ClassPresentationConfig.TierThresholds` grants **+1 combo/action strip slot** for that path. `CharacterProgression.GetClassUpgradeActionSlotBonus` sums reached class tiers across paths; `ComboSequenceMaxHelper.GetEffectiveMax` adds those slots to the loot base plus equipped gear `ExtraActionSlots`, then clamps to `LootSystemConfig.ComboSequenceAbsoluteMax`. Level-up feedback announces a newly gained action slot when the awarded class point crosses a tier threshold.
- **Wizard weapon combo slot:** Any equipped **Wand** (Wizard class weapon) grants **+1 combo strip slot** via `ClassPresentationConfig.WandEquippedComboSlotBonus`, folded into equipment `ExtraActionSlots` totals and shown on weapon tooltips/stats.

### Combat — combo reset at encounter / room / dungeon boundaries

- **Behavior:** Hero (and current enemy) combo routing state resets whenever a combat encounter is initialized, when the player enters a new dungeon room, and when a new dungeon run starts—so the first meaningful combo-path swing of a new fight is never still keyed off the previous encounter’s strip position or combo-mode carryover.
- **Code:** `CombatStateManager.InitializeCombatEntities`, `CombatManager.RunCombat` (start/end uses `ResetCombo` for the player), `RoomProcessor.ProcessRoom`, `DungeonDisplayManager.StartDungeon`; test `CombatStateManagerTests.TestInitializeCombatEntitiesResetsPlayerAndEnemyCombo`.

### Combat — hero debuffs and DoT between encounters

- **Behavior:** After each combat (`CombatManager.RunCombat`), after each survived room (`RoomProcessor`), and when a dungeon finishes or the player exits early (`DungeonOrchestrator`), the hero’s `ClearAllTempEffects()` runs so poison/burn/bleed, stun/weaken/slow/shields, crit-miss penalty, advanced stack effects (vulnerability, fortify, silence, temp HP, etc.), and character-side combo/roll/next-attack buffers do not carry into the next fight or the overworld.
- **Implementation:** `Actor.ClearAllTempEffects` clears every combat status field on `Actor`; `Character.ClearAllTempEffects` still clears `CharacterEffects` and temp stat bonuses. `CharacterFacade.ClearAllTempEffects` delegates to `Character.ClearAllTempEffects` (single path). Tests: `ActorClearTempEffectsTests` in the entity test runner.

### Data / Google Sheets — item suffixes (`StatBonuses.json`)

- **Pull**: `statBonusesSheetUrl` in `SheetsConfig.json` (published CSV or same-doc `edit?gid=…` link as other tabs).
- **Push**: `statBonusesSheetTabName` in `SheetsPushConfig.json` (default `**SUFFIXES`** when the field was missing/blank). Balance Tuning can store the tab **gid** in the new **SUFFIXES → StatBonuses.json** field; push still requires a tab whose **title** matches `statBonusesSheetTabName`.
- **Columns (canonical / JSON):** `Name`, `Description`, `Value`, **`Rarity`** (affix pool tier for loot rolls), `StatType`, optional `ItemRank` (Action Lab catalog filter: blank = visible under every rarity filter), optional **`Requirements`** (attribute thresholds that must be met to equip a piece carrying this suffix). Sheets import still accepts a legacy **`Weight`** column; it is removed and **`Rarity`** defaults to **Common** when no `Rarity` cell was present (`JsonArraySheetConverter.NormalizeStatBonusesJsonArrayRow`). **Runtime:** most rolled suffix `Value` / mechanic entries are a **percentage** of a **reference total** for that stat (e.g. `5` ⇒ +5% of reference STR) in `EquipmentBonusCalculator` when a `Character` context is passed. Positive percentage suffixes have a minimum integer contribution of **+1** when they would otherwise floor to 0. **Literal (flat):** ACCURACY, HIT, COMBO, CRIT, CRIT MISS, RollBonus (sheet ROLL) — `3` means +3, not +3%. **`ALL`** follows the same rule per target stat. Omitting the character context sums all suffix numbers as **flat** (legacy tests).
- **Suffix `Requirements` (bracket cell):** Authored as `[strength:5,primary:15]` — the same bracket / `key:value,…` format used by `Mechanics`. May list one or more entries; **duplicate keys collapse to the maximum** value (so `[strength:5,strength:4]` = `strength 5`). Allowed keys are the four core attributes (`strength` / `agility` / `technique` / `intelligence`) and the four dynamic categories (`primary` / `secondary` / `neglected` / `weakness`) that resolve at equip-check time via `DynamicAttributeCategoryResolver` (e.g. `primary:15` ⇒ "you need 15 in whichever stat is currently your highest"). Unknown keys are dropped on parse. Sheet header aliases `Stat Requirement` / `Requirement` / `Requirements` map onto this field (`JsonArraySheetConverter.RenameStatBonusSheetImportHeaders` + `TryApplyStatBonusBracketRequirements`); the legacy `Stat Requirement` → `ItemRank` mapping was retired since the catalog rarity filter is now authored in the **`ItemRank`** column directly. `StatBonus.Requirements` carries onto each rolled suffix instance (`StatBonus.CloneForItemInstance`); when `LootBonusApplier` finishes adding bonuses, `Item.RecomputeAttributeRequirementsIncludingModifications` folds those suffix gates into the item's `AttributeRequirements` (per-stat **max** across catalog, prefix, and suffix sources) so equip checks, the red-bracket UI, and the `Requires:` tooltip line stay consistent. Tests: `ItemTests` (suffix gate merge + dynamic category equip-block), `JsonArraySheetConverterTests` (bracket import).

### Action Interaction Lab — controls pop-out layout
