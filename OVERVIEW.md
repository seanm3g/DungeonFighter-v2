# DungeonFighter v2 — Overview

DungeonFighter v2 is a turn-based RPG written in C# (.NET) with a data-driven combat/action system, character progression, and an Avalonia UI. Gameplay is built around **actions** (attacks, heals, effects) that are selected and executed in a structured combat flow, with systems for combo routing, thresholds, roll modifiers, and post-hit processing.

## Core gameplay pillars

- **Turn-based combat**: Entities take turns selecting and executing actions.
- **Actions as data**: Most combat behavior is configured through JSON/spreadsheet-driven action data that maps into runtime `Action` objects.
- **Combo system**: Actions can chain; combo routing can jump/skip/repeat via `ComboRouter`. Each **weapon type** (Sword, Dagger, Mace, Wand) has one **required basic** attack that must appear in the hero’s combo sequence while that weapon is equipped: it is defined in `Actions.json` with tag **`weapon_basic`** and a matching **`weaponTypes`** entry (`WeaponRequiredComboAction`, `ComboSequenceManager` ensure/inventory removal rules). Canonical basics: **STRIKE** (Sword), **STAB** (Dagger), **SLAM** (Mace), **CAST** (Wand); if no tagged row is found, those names are used as fallbacks. **`GearActionNames.Resolve`** always includes that required basic in the weapon’s resolved action-name list when absent (so starter/loot weapons that only list e.g. **JAB** still load **STRIKE** into the pool and the cannot-remove / re-inject rules apply). **Intelligence (effective INT) and the strip:** at **`GameConstants.ComboSequenceIntelligenceThreshold`** (10) or higher, which combo-strip action fires on a combo-path roll follows **`ComboStep`** in order, and **`ComboRouter`** rules apply after each successful combo hit. Below that threshold, the strip slot is **chosen at random** among configured combo actions (heroes and enemies), and **`ComboStep`** advances **only to the next sequential slot**—authored **`ComboRouter`** branches (jump/skip/repeat/stop, etc.) are ignored (`Character.IncrementComboStep`). The combat strip highlight still tracks `ComboStep`, so it may not match a chaotic swing until INT reaches the threshold. If the strip has **exactly one** combo action, chaotic selection does not roll dice for the slot (`ActionUtilities.ResolveComboStripIndex` returns 0)—otherwise a `1d1` would be invalid and could crash combat at the first hero action roll. **Below combo threshold (named combo-strip swing vs synthetic normal):** the modified **d20** from the die (sheet roll modifications to the face only; not STR/INT/etc. roll bonuses) must meet the configured combo threshold (default 14; COMBO-type queued adjustments can still change that number), except on a natural 20. Queued **ACCURACY** shifts hit and post-roll **combo outcome** thresholds in combat, but does **not** lower the bar used to pick a named combo-strip action—so a low die cannot read as **roll: 9** yet execute a **14+** combo special. Otherwise `ActionSelector` uses an **unnamed synthetic normal** hit for both heroes and enemies—including enemies whose action pool is **only** combo-flagged specials, so they no longer fall through to the first special at low rolls. **Encounter boundaries:** `CombatStateManager.InitializeCombatEntities` calls **`Character.ResetCombo()`** on the hero and current foe so each new fight starts from the opener (step 0, combo mode off, baseline amp state); dungeon **room entry** (`RoomProcessor`) and **new dungeon** (`DungeonDisplayManager.StartDungeon`) also reset the hero’s combo so strip/UI never carries a prior room’s step into exploration.
- **Advanced mechanics**: Thresholds, roll modification, status effects, and multi-hit processing participate in the action execution pipeline. **Combat HUD thresholds** (left/right panels) preview queued FIFO/slot **ACCURACY** / **HIT** / **COMBO** shifts only; persistent stat/equipment roll bonus is **not** folded into those rows (it is added to attack total in `ActionExecutionFlow` / `HitCalculator`). Large deferred accuracy (e.g. wand **CAST** `heroAccuracy`) can still lower a *raw* preview below 1; the UI clamps the displayed ladder number to **1** so it stays readable, matching the minimum **1** floor applied when resolving effective threshold values for display.
- **Settings-driven tuning**: The settings UI persists changes to `GameSettings` and applies some updates immediately (e.g., action pool refresh) while other settings apply on next use.

## Key systems (high-level)

- **Google Sheets ↔ JSON (weapons, mods, armor, suffixes, enemies, environments, dungeons)**: [`GameData/SheetsConfig.json`](GameData/SheetsConfig.json) can list published CSV URLs per tab (`weaponsSheetUrl`, `modificationsSheetUrl`, `armorSheetUrl`, **`statBonusesSheetUrl`** for item suffixes → **`StatBonuses.json`**, plus `enemiesSheetUrl`, `environmentsSheetUrl`, `dungeonsSheetUrl`). `PULL_SHEETS` / `GameDataSheetsPullService` writes the matching JSON files via [`JsonArraySheetConverter`](Code/Data/JsonArraySheetConverter.cs) (one header row per tab except ENEMIES; any extra column becomes JSON). **`PUSH_SHEETS`** uses [`GameData/SheetsPushConfig.json`](GameData/SheetsPushConfig.template.json) tab names (`weaponsSheetTabName`, …, **`statBonusesSheetTabName`**); if **`statBonusesSheetTabName`** is blank on an older config, push defaults it to **`SUFFIXES`**—your spreadsheet tab title must match that name, or change the config to your exact tab title (the API uses the tab name, not `gid=`). **SUFFIXES tab** (→ `StatBonuses.json`): columns **`Name`**, **`Description`**, **`Value`**, **`Weight`**, **`StatType`** (same shape as [`GameData/StatBonuses.json`](GameData/StatBonuses.json)). If enemies/environments were left blank (common on older configs that only set weapons/armor), the game defaults them to **`ENEMIES`** and **`ENVIRONMENTS`**; if **`DUNGEONS`** is blank it defaults the same way so local `Dungeons.json` can upload—edit the config if your tab titles differ. **DUNGEONS tab** (→ `Dungeons.json`): `name`, `theme`, `minLevel`, `maxLevel`, `possibleEnemies` (JSON string array or pipe-separated enemy names matching **`Enemies.json`**), optional `colorOverride` (JSON object cell). **ENEMIES tab** canonical columns: `name`, `archetype`, `baseAttributes.{strength,agility,technique,intelligence}`, `growthPerLevel.{same four}`, `baseHealth`, `healthGrowthPerLevel`, `actions`, `isLiving`, `description`, `colorOverride` (JSON object cell when used). **Growth budget:** STR+AGI+TEC+INT growth per level is normalized to sum **6** in `EnemyStatCalculator` (fractional weights OK). Legacy **`overrides`** JSON (especially `health`) is still accepted on import and folded into `baseHealth` / tuning. Import still accepts a legacy single **`baseAttributes`** / **`growthPerLevel`** column as JSON; dotted columns merge on top. **`actions`** may be pipe-separated (`JAB|TAUNT`) and are normalized to a JSON array on import. **ENVIRONMENTS tab** (→ `Rooms.json`): `name`, `description`, `theme`, `isHostile`, `actions` (JSON array of `{name,weight}`), optional **`enemies`** (same shape—weighted enemy template names from `Enemies.json`). When `enemies` is non-empty for a room, hostile generation uses that pool (weighted) and ignores the dungeon’s `possibleEnemies` filter for template choice.
- **Enemy attributes from `Enemies.json`**: `baseAttributes` and `growthPerLevel` use nullable numeric fields per stat. Shipped enemy rows list **all four** base combat stats (STR/AGI/TEC/INT) so the ENEMIES sheet and HUD stay in sync with designer intent. If a row still omits keys, omitted stats are **not** treated as zero; `EnemyStatCalculator` fills them from tuning **baseline × archetype**. Per-level attribute gains come only from **`growthPerLevel`**, normalized so STR+AGI+TEC+INT growth sums to **6** each level. Explicit `0` in JSON still means zero for that cell when present.
- **ENEMIES sheet column layout (canonical, 16 columns A–P)**: **A** `name`, **B** `archetype`, **C–F** base attributes (`strength`, `agility`, `technique`, `intelligence`), **G–J** growth per level (same four), **K** `baseHealth`, **L** `healthGrowthPerLevel`, **M** `actions`, **N** `isLiving`, **O** `description`, **P** `colorOverride`. Row 1 is the **category band** (`base attributes`, `growth`, …); row 2 is the short field names (see `JsonArraySheetConverter`). Older `Enemies.json` rows sometimes had extra **root** keys (`strength`, `agility`, …); those fold into **`growthPerLevel`** on load (`EnemyDataLegacyRootStats`) and when importing CSV (`HoistEnemyLegacyRootNumbersIntoNested`), and push maps them into the canonical block instead of trailing extras.

- **Actions spreadsheet (`GameData/Actions.json`)**: Rows can be tagged for **non-hero** pools. Use **category** `ENEMY` with **tags** including `enemy` for enemy-only attacks (so loot/weapon resolution does not assign them to the hero). Use **category** `ENVIRONMENT` with **tags** including `environment` plus a **lowercase dungeon theme** (e.g. `forest`, `crypt`); optional `room-{type}` in tags scopes data. Environment hazards use spreadsheet **target** `AOE`. Runtime room loading filters environment actions by theme (or the `generic` tag). Environmental rows use the legacy `EnvironmentalActions.json` **id** as the spreadsheet **action** key (**uppercase**, words separated by spaces; room-scoped legacy ids omit the old `room_` prefix) so every row stays unique.

- **Settings system**: Avalonia settings panels save through an orchestrator and apply via a single apply service. See `Documentation/05-Systems/SETTINGS_SYSTEM_ARCHITECTURE.md`.
- **Inventory (Avalonia)**: On the **inventory** screen, the top **action strip** mirrors the hero’s combo sequence (same order as the right-hand SEQUENCE panel). **Right-click** a filled strip card to remove that slot—same rules as removing from the sequence list (`InventoryComboManager` / `cpi:rm:N`, including **weapon-required** basic protection). Drag reorder on the strip remains available when combo reorder is allowed (not during an active dungeon run). Equip / compare / trade-up prompts ignore strip removes until you finish or cancel those flows.
- **Action execution**: Action execution is orchestrated and delegated to specialized executors and processors (attack/heal, multi-hit, etc.). See `Documentation/01-Core/ARCHITECTURE.md` for file entry points.
- **Character effects**: Character/actor state includes transient “next action” modifiers (speed/damage/amp/multihit) that should affect the following action execution.

### Poison, burn, and bleed (DoT)

- **Poison**: Cumulative **percent of max HP** (`PoisonPercentOfMaxHealth`). Successful applications **add** to that %; the global DoT tick deals `floor(maxHP × % / 100)` and **does not** reduce stored %. **Cleanse** can shave stored % by a small fixed step. Tuning uses a shared **5s game-time** tick interval (`poison.tickInterval` in `TuningConfig.json`) with burn. **Weapon `weaponPoison` mods** roll a per-crit % from each mod’s `MinValue`/`MaxValue` in `Modifications.json` (e.g. Venomous **1–2%**, Venomous Blade **2–3%** on a **critical hit** only); those rolls sum for the delta applied on that crit. Poison, burn, and bleed from **weapon stats** or weapon **`StatusEffects`** apply only when `CombatEvent.IsCritical` is true. **Actions** with built-in `CausesPoison` / `CausesBurn` / `CausesBleed` and **no** spreadsheet **trigger conditions** follow the same crit-only rule; if the action defines **trigger conditions** (`ONHIT`, `ONCRITICAL`, etc.), those conditions gate the whole status bundle as before. Application runs once inside `ActionExecutionFlow`; the colored UI path must only **display** `StatusEffectMessages` from that run (re-applying would double the debuff).
- **Burn**: **Intensity** plus **pending from hits**; each global tick deals damage equal to current intensity, then decays intensity by 1 and merges pending (same pattern for processing order in code).
- **Bleed**: Same **intensity + pending** model as burn, but **one pulse when the afflicted actor finishes their combat turn** (not on the global poison/burn tick). Weapon mods use deterministic **`weaponPoison` / `weaponBurn` / `weaponBleed`** values on **critical hits** only (no random proc rolls for those).
- **Combat log spacing**: Global poison/burn tick output is recorded as a `PoisonDamage` UI block with the **afflicted** actor’s name. `TextSpacingSystem` uses that so a tick on the hero does not insert an extra blank line before the hero’s following combat action (the prior line’s “last attacker” would otherwise be the enemy).

## Current focus area (for this change)

### Progression — XP to next level and mid-run level-ups

- **XP bar** (`CharacterProgression.GetXpRequiredToAdvanceFromLevel`): one **tier-1 dungeon completion** (same 10× formula as `XPRewardSystem.AwardDungeonCompletionXP` at dungeon level **1**) is the unit. The bar is **that many completions**: **1** for L1→2 (so the first dungeon run finishes a level if completion XP is the main grant), then **1.5**, **2**, **3**, **4**, **5**, … for higher levels so it takes more runs to level over time. `Progression.BaseXPToLevel2` &gt; **0** scales the whole curve so the L1→2 bar equals that value; `Progression.XPScalingFactor` (when &gt; **0**) multiplies every bar as a global pace knob (≤ **0** ⇒ **1.0**). Helpers: `GetBaseXpForContentLevel`, `GetStandardDungeonCompletionXpForLevel`, `GetExpectedDungeonsToLevelFromLevel`.
- **Immediate XP** (`CharacterFacade.AddXP`): progression already increments `Level` per threshold; reward application uses `LevelUpWithInfo(levelAlreadyIncremented: true)` and passes the **per-step** level (`oldLevel + step + 1`) for messages, dungeon completion entries, and class points so multi-threshold grants stay consistent.

### Combat — combo reset at encounter / room / dungeon boundaries

- **Behavior:** Hero (and current enemy) combo routing state resets whenever a combat encounter is initialized, when the player enters a new dungeon room, and when a new dungeon run starts—so the first meaningful combo-path swing of a new fight is never still keyed off the previous encounter’s strip position or combo-mode carryover.
- **Code:** `CombatStateManager.InitializeCombatEntities`, `CombatManager.RunCombat` (start/end uses `ResetCombo` for the player), `RoomProcessor.ProcessRoom`, `DungeonDisplayManager.StartDungeon`; test `CombatStateManagerTests.TestInitializeCombatEntitiesResetsPlayerAndEnemyCombo`.

### Data / Google Sheets — item suffixes (`StatBonuses.json`)

- **Pull**: `statBonusesSheetUrl` in `SheetsConfig.json` (published CSV or same-doc `edit?gid=…` link as other tabs).
- **Push**: `statBonusesSheetTabName` in `SheetsPushConfig.json` (default **`SUFFIXES`** when the field was missing/blank). Balance Tuning can store the tab **gid** in the new **SUFFIXES → StatBonuses.json** field; push still requires a tab whose **title** matches `statBonusesSheetTabName`.

### Action Interaction Lab — controls pop-out layout

The **Action Lab** auxiliary window lists the current **foe**, **HP**, a scrollable **enemy type** picker (▲ types / ▼ types), then turn info (**Next** actor, **Actions taken** counter, strip step), d20 grid, and the scrollable **action catalog**. **Actions taken** increments for each interactive **[ Step ]** that records a lab turn (same as undo history), adds the sum of resolved combat turns across encounters after a **[ Sim N ]** batch finishes, and resets when the lab clears fight history (gear apply/clear, enemy type swap, enemy level rebuild, or the tools-panel **[ Reset ]** control). **[ Reset ]** also clears the center combat log, refills both fighters to full HP, clears status and temporary combat effects, and returns both combo strip pointers to the first slot while **keeping** equipped items, the combo strip contents, and the current enemy encounter. **Back** (undo) removes the last interactive step from that count. The enemy-type list shows **five** visible rows (was two) so more foes are visible at once; the catalog uses the remaining vertical space above the fixed footer.

When the lab starts, window **placement** (sizes unchanged) uses the monitor that contains the main game window: the main window is **centered** in that display’s **working area**, and the lab tools window is **right-aligned** inside the same working area with a small margin so it stays on-screen on smaller displays. If a pop-out **Game Settings** window is already open on that same monitor, it is **nudged to the left** inside the working area so the three-window layout matches a typical wide-desktop arrangement (`ActionLabWindowPlacement`, `ActionLabControlsWindow`).

### Combat right panel — location names

In the **LOCATION** block, **Dungeon** and **Room** values (and the enemy name line when combat is active) ellipsize only when wider than the panel’s inner monospace width (`LayoutConstants.RIGHT_PANEL_WIDTH` minus content inset), not a fixed 20-character cap—so names like **Tomb of the forgotten** render in full.

### Combat right panel — status effects (enemy)

The **right panel** status-effects block labels **Enemy:** active DoT/debuff lines from `StatusEffectDisplayLines`. When an enemy template is **non-living** (`isLiving` false in `Enemies.json` → `Enemy.IsLiving`), the game does not apply **bleed** or **poison % DoT** ticks to that foe; the panel shows a cyan line **Immune: Bleed, Poison** under **Enemy:** so it matches combat rules (distinct from **IsAlive**, which is HP > 0).

### Action Interaction Lab — gear editor (right-click equipment)

Right-clicking a gear row in lab combat opens the **Action Lab — weapon / armor** dialog. You pick a base row from data, then **multiple** optional **prefix** rows (modifications) and **multiple** optional **suffix** rows (stat bonuses); each list is multi-select (toggle rows on or off). Apply builds the item with every selected prefix and suffix (rolled mod values still use each template’s min/max range). The dialog opens at about **1000×560** client size so lists are not cramped, is **resizable**, and the three scrollable lists **split extra height equally** so you see more rows without only tiny fixed caps. The **weapon** dialog adds **Weapon type** and **Rarity** dropdowns: type narrows the weapon list (Mace, Sword, Dagger, Wand); rarity narrows **prefixes** by each row’s `ItemRank`, **suffixes** by optional `ItemRank` on `StatBonuses.json` entries (blank `ItemRank` keeps a suffix visible for every rarity), and **weapons** by a tier→rarity band (T1 Common … T5 Legendary, with **Mythic** / **Transcendent** still showing tier-5 bases). **Head, body, and feet** use the same rarity behavior on base armor, prefixes, and suffixes, plus an **Armor class** dropdown: classes are the armor display name with the last word removed (e.g. `Cloth Cap` → `Cloth`, `Studded Leather Boots` → `Studded Leather`), built from the rows in `Armor.json` for that slot. The SUFFIXES sheet / JSON canonical columns include optional **`ItemRank`** alongside `Name`, `Description`, `Value`, `Weight`, `StatType`.

### Action Interaction Lab — enemy level (right panel)

The **Action Interaction Lab** is a stepped combat sandbox (cloned hero, lab enemy, forced catalog action, fixed or random d20). The combat **right panel** shows the current foe (HP, combat stats, thresholds). This change adds:

- **Enemy level** on the right panel in lab combat (`Lvl N` plus **(Δ)** vs the lab hero: `Lvl 9 (-2)` when the hero is 11; Δ = enemy level − hero level), styled like the hero’s level on the left.
- **Hero level** (left panel `Lvl`): left-click / right-click adjusts the hero by one step as before, and applies the **same** level delta to the lab enemy when the hero’s level actually changes (clamped 1–99), so both stay aligned from the hero row. **Enemy level** on the right panel still changes **only** the enemy (independent of the hero).
- **Left / right click** on the right-panel **enemy** level row to raise or lower enemy level (1–99) alone. The lab **rebuilds** the enemy at the new level (loader enemies via `EnemyLoader.CreateEnemy`; default dummy via `TestCharacterFactory` with per-level HP/damage/armor/speed scaling), then **clears step history** and **re-bootstraps** combat so encounter state stays consistent.
- **Undo / replay** and **batch simulation** snapshots carry the effective enemy level (`LabCombatSnapshot.EnemyLevel`, `CaptureSimulationSnapshot`).
- **Direct-stat dummies**: `Enemy.GetEffectiveStrength()` returns base **Damage** (so damage math and the DMG line match); the panel also shows **attack speed**.

### MultiHit Mod (Hero base stats → next action) — prior milestone

The settings UI exposes **Hero base stats** including action speed %, action damage %, **MultiHit mod**, and amp mod %. Those modifiers apply to the hero’s **next** executed action; prior work ensured MultiHit mod flows through data, execution, and tests.
